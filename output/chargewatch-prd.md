# ChargeWatch — 产品需求文档 (PRD)

## 1. 产品定位

**ChargeWatch** 是一款只做一件事的 macOS 菜单栏小工具：**让你随时看到 Mac 实时充电功率，并完整记录历史**。

- **目标用户**：自己（chenran）— 关心 Mac 实际从充电器拉了多少瓦、电池是不是真的在充电、不同场景下的功耗模式
- **使用场景**：
  - 换新充电器/线材时验证 PD 协商是否到位（说好 100W，实际只拿到 60W？）
  - 高负载工作时判断电是不是入不敷出（系统功耗 > 适配器输入 = 电池在帮忙供电）
  - 长期回看「今天充了多少电、什么时段在充」
  - 出差时排查酒店墙插充电速度

## 2. 核心价值主张

| 维度 | 现状（不用本工具） | 用本工具后 |
|---|---|---|
| 实时功率 | 看不到，只知道"在充电" | 菜单栏一眼看到 67.2 W |
| 充电诊断 | 慢了不知道哪里慢 | 看到适配器实际输入 vs 额定，立刻定位 |
| 历史回溯 | 无 | 任意时段查询、CSV 导出 |
| 入侵感 | — | 仅菜单栏一个图标，无 Dock、无窗口干扰 |

## 3. 功能范围（MVP）

### 3.1 必做（P0）
1. **菜单栏实时显示**
   - 充电中：图标（SF Symbol `bolt.fill`）+ 数字瓦数，如 `⚡ 67W`（图标是 SF Symbols，不是 emoji）
   - 未充电但接电：图标（`bolt.slash`）+ `AC`
   - 纯电池放电：图标（`battery.50percent`）+ 当前 SoC%
   - 无电池（桌面 Mac）：图标（`powerplug`）+ 系统功耗
2. **下拉详情面板**（点击菜单栏图标弹出）
   - 当前状态横幅：充电中 / AC 未充 / 放电中
   - 关键数字（4 项）：充入电池功率、适配器实际输入、系统总功耗、电池 SoC
   - 适配器信息：额定瓦数 + 协商电压电流（如 100W PD, 20V/5A max）
   - 充电曲线：最近 1 小时 sparkline，hover 显示具体数值
   - 操作入口：「查看完整历史」「导出 CSV」「设置」「退出」
3. **数据采集与持久化**
   - 1Hz 采样写入 SQLite
   - 自动降采样（24h/7d/30d 三档）
   - 跨进程重启不丢数据
4. **历史窗口**
   - 时间范围切换：今天 / 本周 / 本月 / 自定义
   - 折线图：功率 vs 时间，可叠加 SoC 副坐标
   - 统计卡片：累计充入 (Wh)、平均充电功率、峰值功率、充电时长

### 3.2 应做（P1）
5. **CSV 导出**：选定时间范围导出全量字段
6. **开机自启**：通过 `SMAppService` 注册
7. **深色 / 浅色模式适配**
8. **充电完成通知**（可关）：电量到 100% 触发系统通知

### 3.3 可做（P2，后续迭代）
9. 自定义充电达到阈值通知（如 80% 提醒拔线）
10. 多日对比视图
11. 异常检测：检测到适配器实际输入显著低于额定时弹提示
12. 数据备份/恢复

### 3.4 不做（Out of Scope）
- SMC 软限位式充电控制（AlDente/batt 的领域：root 特权助手 + 充电抑制键，与系统原生上限并行冲突）。注：v0.4 新增的"充电上限调节"走的是与系统**同步**的 App Intents 路径，**不**属于此项排除，详见 §9。
- 电池健康度详尽分析（coconutBattery 的领域）
- CPU/GPU/网络监控（Stats / iStat Menus 的领域）
- iOS 设备电量（Juicy 的领域）
- iCloud 同步、多设备账号

## 5. 成功度量

- 启动到菜单栏显示首个真实数据 < 1 秒
- 持续运行 7 天内存 < 50 MB
- CPU 空闲占用 < 0.5%
- 数据库 30 天内总占用 < 50 MB
- 0 个由本应用导致的系统崩溃 / 通知

## 6. 约束与依赖

- **平台**：macOS 13.0+（用 `SMAppService` 需 13+）
- **架构**：Apple Silicon 优先，Intel 降级
- **签名**：本地使用，无需 Apple Developer 账号，本地 ad-hoc 签名
- **App Store**：**不上架**（因使用 IORegistry 私有 key）
- **依赖**：仅 GRDB.swift（SQLite ORM），其余全部使用系统框架
- **语言**：Swift 6 + SwiftUI（macOS 13+）

## 7. 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| AppleSmartBattery 字段在未来 macOS 版本变化 | 数据缺失 | 字段访问全部 optional，缺失时降级到 IOPS API |

## 9. 充电上限调节（v0.4 新增）

### 9.1 目标
让用户在 ChargeWatch 菜单面板内直接调节电池充电上限（80%–100%，5% 步长），并**与系统设置（电池 → 充电 → 充电上限）保持同步**，省去进系统设置层层点击的麻烦。

### 9.2 可行性结论（详见 `output/chargewatch-charge-limit-research.md`）
- **读取（已验证、真同步）**：`pmset -g battlimit` 按 `chargeSocLimitReason = manualChargeLimit` 甄别用户上限，实时反映系统当前值，无需 root。
- **设置（已确认动作存在、路径待 spike 定型）**：经 macOS 26.4 新增的 App Intents 动作 `SetBatteryChargeLimitAction`（写入与系统滑块同源的 powerd 值）+ 用户一次性快捷指令 + `/usr/bin/shortcuts run`。无需 root。**三项关键未验证假设**（编码前 spike 硬门禁，见架构 §13.9）：`shortcuts run` 是否后台静默不前置 Shortcuts App；非沙盒 `Process` 调用是否触发"自动化"(TCC) 授权及被拒行为；单指令能否动态传百分比。任一不达标则该路径降级。
- **放弃** SMC 软限位（不同步、需 root、冲突、不可上架），见 §3.4。

> **同步方向的诚实表述**：app→系统 为实时写入（设置即改系统值）；系统→app 仅在**菜单面板打开时**回读刷新（菜单栏图标本身不显示上限）。不宣称"任何时刻双向实时一致"。

### 9.3 功能范围
- P0：菜单面板显示**当前充电上限**（"80%" / "未设上限"），随系统设置变化实时更新。
- P0：80/85/90/95/100 五档调节控件；选择后调用桥接快捷指令设置，并回读 `pmset` 确认。
- P0：能力门禁——仅当 macOS ≥ 26.4 且为 Apple Silicon 时显示"调节"控件；否则仅显示当前上限 + "在系统设置中调节"深链。
- P0：桥接快捷指令缺失时的一次性引导（onboarding），并始终提供"打开系统设置电池页"兜底深链。
- 不做：<80% 的限位；"仅今天"临时上限（本期固定持久生效）；SMC 强制限位。

### 9.4 用户故事
| ID | 故事 | 验收标准 |
|---|---|---|
| US-07 | 作为用户，我想在菜单里一眼看到当前充电上限 | 面板**打开时**显示当前上限；系统设置改动后，下次打开面板（或面板打开期间 ≤15s 轮询）刷新一致 |
| US-08 | 作为用户，我想在菜单里直接把上限调到 80/85/90/95/100 | 选择后系统上限随之改变，退避回读 `pmset -g battlimit` 一致（≤5s 内）；**前提是 §13.9 spike 通过**，否则降级为只读+深链 |
| US-09 | 作为首次使用者，我需要被引导完成一次性快捷指令配置 | 未检测到桥接指令时弹出引导，配置后调节即生效 |
| US-10 | 作为旧系统/不支持机型用户，我至少能快速跳到系统设置 | 不支持时显示当前上限（若可读）+ 一键直达系统设置电池页 |

### 9.5 约束
- 仅 Apple Silicon（按硬件探测，非进程架构）+ macOS 26.4+ 支持"app 内设置"；读取与深链在支持原生上限的系统上可用。
- 依赖系统二进制 `/usr/bin/pmset`、`/usr/bin/shortcuts`，经 `Process` 调用（非沙盒构建，ad-hoc 签名）；Info.plist 需 `NSAppleEventsUsageDescription`。
- 无程序化创建快捷指令的 API → 需用户一次性配置（尽量做成最少步骤 / 清晰引导）。
- 关键路径假设以编码前 spike（架构 §13.9）为硬门禁；不达标则相应降级，不强行交付不可靠功能。
- 本期段控仅 80/85/90/95/100；"清除上限/回到无限制"是否可从 app 内完成取决于动作语义（spike 验证），否则只读显示 + 深链清除。
